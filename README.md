# LiverPool AI

ИИ-платформа проактивного скрининга заболеваний печени и хронических вирусных
гепатитов. Кейс №04, Astana AI Week 2026, трек Healthcare.

Печень не имеет болевых рецепторов — фиброз и цирроз годами развиваются
незаметно для пациента. LiverPool AI считает риск по обычным анализам крови,
которые пациент и так сдаёт (FIB-4, APRI + XGBoost с SHAP-объяснением),
отслеживает тренд показателей за 5 лет и сортирует пациентов по срочности —
без единого нового теста. Отдельно система показывает валидацию модели
сегментации КТ-снимков печени (датасет LiTS) и валидацию табличных моделей
на реальных, независимо опубликованных клинических датасетах (UCI Hepatitis C,
Mayo Clinic PBC) — чтобы честно показать, что подход генерализуется за
пределы синтетической когорты, а не только выдаёт красивые цифры на
сгенерированных пациентах.

Демо-видео: см. ссылку в описании хакатон-заявки.

## Структура проекта

```
liverpool-ai/
├── backend/            — FastAPI + ML (Python)
│   └── app/
│       ├── data/        — генератор синтетической когорты, врачи/участки,
│       │                  лабораторные документы, реальные датасеты (CSV)
│       ├── models/      — риск-движок (XGBoost + SHAP), модели на реальных
│       │                  данных (Hepatitis C, Mayo PBC), CV-модель
│       └── main.py      — API-эндпоинты
├── frontend/            — Next.js (TypeScript)
│   ├── app/              — страницы (кабинет врача, онко-скрининг,
│   │                        валидация модели КТ, статистика клиники)
│   ├── components/
│   └── lib/
└── research/            — исходники разработки моделей (не часть работающего
                            приложения, но показывают, как получены метрики,
                            которые видны в интерфейсе)
    ├── ct-segmentation-model/            — обучение и валидация 3D UNet для
    │                                       сегментации печени/опухолей на КТ
    │                                       (датасет LiTS, 51 случай)
    └── hepatitis-c-real-data-model.ipynb — пайплайн XGBoost на реальном
                                             датасете UCI Hepatitis C
```

## Как запустить

Нужны: **Python 3.11+** и **Node.js 20+**.

### 1. Backend

```bash
cd backend
python3 -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate

pip install -r requirements.txt  # torch включён в requirements.txt, отдельно ставить не нужно

uvicorn app.main:app --host 0.0.0.0 --port 8000
```

При первом запуске backend сам сгенерирует синтетическую когорту (1800
пациентов, калиброванных под опубликованную эпидемиологию Казахстана) и
закэширует её в `backend/app/data/cohort.pkl` — это займёт несколько секунд.
Обученные модели (риск-движок, модели на реальных датасетах) уже включены в
репозиторий.

Backend поднимется на `http://localhost:8000`. Проверить:
`curl http://localhost:8000/api/health` должен вернуть
`{"status":"ok","patients_loaded":1800}`.

### 2. Frontend

Открыть **новый терминал**, backend должен продолжать работать:

```bash
cd frontend
npm install
npm run dev
```

Сайт откроется на **`http://localhost:3000`** — это и есть интерфейс, который
нужно открыть в браузере. (`localhost:8000` — это только API, там нет
страницы для просмотра, это нормально.)

## Вход

На `/login` — демо-вход: выбор врача из списка (8 врачей, каждый закреплён за
своим участком, ~200–250 пациентов), без пароля. Это осознанное ограничение
прототипа, явно описанное в интерфейсе (см. «Известные ограничения» ниже).

## Что смотреть в интерфейсе

| Раздел | Что показывает |
|---|---|
| Кабинет врача (`/`) | Список пациентов своего участка с авто-расчётом риска по обычным анализам крови; карточка пациента — тренд анализов за 5 лет, SHAP-объяснение, лабораторные документы, направление к гепатологу, КТ-данные (если есть у пациента) |
| Онко-скрининг (`/oncology-screening`) | Все пациенты с ненулевым % онкологического риска, отсортированы по убыванию — кого направлять на углублённое обследование в первую очередь |
| Валидация модели (`/imaging`) | Витрина: 51 случай из тестовой части датасета LiTS (сегментация ИИ vs врачебный консилиум), честно вычисляемые Accuracy/Sensitivity/Specificity, плюс валидация табличных моделей на реальных датасетах (Hepatitis C, Mayo PBC) |
| Статистика клиники (`/analytics`) | Агрегированная статистика по всей когорте для главного врача/отчётности + песочница «что если увеличить охват скринингом» |

## Известные ограничения (честно, для показа жюри/докторам)

- **Данные — синтетическая когорта**, калиброванная под опубликованную
  эпидемиологию Казахстана, а не реальные пациенты. Модели на реальных
  датасетах (Hepatitis C, Mayo PBC, LiTS) валидируются отдельно, честно и
  без приукрашивания метрик.
- **Демо-вход без пароля**, и backend не проверяет на сервере, что врач
  запрашивает данные именно своего участка — только клиентская фильтрация.
  Для реального пилота нужна настоящая аутентификация и серверная
  авторизация.
- **Направления и обратная связь по SHAP-объяснению хранятся в памяти
  процесса** — переживают перезагрузку страницы, но не переживают
  перезапуск backend. Реальному пилоту нужна постоянная БД.
- **КТ-показатели** — это точный пересчёт по 51 случаю датасета LiTS,
  а не загрузка живого DICOM-файла конкретного пациента.
- Интерфейс только на русском языке.

## Если что-то не работает

- **Backend не отвечает** — проверить, что оба терминала (backend и
  frontend) открыты и не упали; `curl http://localhost:8000/api/health`
  должен ответить.
- **Порт занят** — если 3000 или 8000 уже используются, остановить процесс
  на этом порту или указать другой (`uvicorn app.main:app --port 8001`, и
  поменять `NEXT_PUBLIC_API_URL` в `frontend/.env.local` соответственно).
